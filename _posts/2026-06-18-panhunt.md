When I started working in a payment processor, PANhunt was already in use on many servers. It was not a tool I discovered later because I wanted to write something around PCI DSS, and it was not a side project looking for a use case. It was already part of a practical routine: search systems for possible primary account numbers and help validate whether PAN existed outside the places where we expected it to exist. That distinction matters, because a tool looks very different when it is already in operational use. The question is no longer whether the idea is useful. The question becomes whether the implementation survives the data it is supposed to inspect.

PANhunt had the right basic shape. It searched files and containers for card-number patterns, validated candidates with Luhn checksum logic, and reported masked values instead of full PANs. For a PCI environment, that is a useful control-supporting function. It does not prove compliance, and it does not replace data-flow analysis, application review, database inspection, segmentation validation, retention governance, or operational discipline. But it answers a specific question that still matters in payment environments: can we find clear-text PAN-like values in places where they should not be?

The uncomfortable part was not the pattern matching. The uncomfortable part was the input. Real environments contain compressed logs, old exports, nested archives, mailbox files, support bundles, forgotten working directories, document containers, and files produced by systems that no longer have a clear owner. Some of that content is large. Some of it is malformed. Some of it is nested. Some of it expands far beyond what its stored size suggests. A compressed file can be larger than memory after decompression, and a zip bomb is only the hostile version of a problem that can also appear accidentally in ordinary operational data. If a scanner assumes that compressed content can be safely expanded into memory, the scanner itself becomes the operational risk.

That was the pressure behind PANhunt 2.0. The problem was not merely to make the code cleaner, although the code did need cleaning. The problem was to make the act of searching more bounded. PAN discovery is not only a regular-expression problem. Once the tool enters archives, compressed streams, mail stores, attachments, PDFs, Office files, and OpenDocument containers, it becomes a resource-management problem as well.

PANhunt is not originally my project. It comes from the Dionach PANhunt project, and the original BSD-3-Clause license and copyright notice remain preserved. I kept the attribution visible because the original tool had practical value and that should not disappear behind a fork. Dionach made a small, useful PAN discovery utility available to the PCI and security community. My work has been to keep that utility useful under the conditions where I saw it being used: not in a clean demo folder, but against messy server data where recursive scanning has to be treated carefully.

PANhunt 2.0 is almost a total rewrite. That was not the original ambition, but it became difficult to avoid once the real problem was clear. The issue was not isolated to one parser, one archive handler, or one command-line option. Compressed files, recursive containers, attachment handling, configuration, reporting, worker dispatching, and parser safety all touched each other. Fixing only one part would have left the same structural weakness elsewhere.

The rewrite moved the project toward clearer boundaries. `PanHuntService` now orchestrates a scan without terminal-output concerns. `CliPresenter` handles output and report writing. `ScanConfiguration` carries resolved configuration into the scan rather than relying on implicit global state. `Hunter` and `Dispatcher` handle traversal and concurrent work. Scanner and archive factories select the relevant parser or container handler. Findings and scan results are carried as structured objects instead of existing only as printed output.

This was not cleanup for its own sake. A recursive discovery tool needs predictable boundaries because its input is unpredictable. If configuration is implicit, if scanner behaviour depends on hidden state, or if output and scanning are mixed together, the tool becomes harder to test and harder to reason about. That matters when the tool is walking through directories, opening archives, scanning attachments, parsing documents, and reporting sensitive-data findings. The more uncontrolled the input is, the less tolerance there should be for uncontrolled internal behaviour.

The compressed-file issue became the clearest example. It is tempting to treat archive handling as a simple parsing step: open the archive, extract the children, scan them, and continue. That works until the archive is too large, too nested, too compressed, too malformed, or too expensive to process. In PANhunt 2.0, archive handling is treated as a budgeted operation. The scanner now has configurable limits for expanded size, archive member count, compression ratio, path length, attachment size, attachment count, total attachment bytes, parser timeout, parser memory, PDF pages, PDF text size, scan depth, child jobs, and total expanded bytes.

Those controls are not there to make the configuration file look impressive. They exist because the tool should be able to refuse unsafe work. If a compressed input exceeds the configured expansion budget, that is not just a scan failure. It is information. It tells the operator that this object needs a different handling path, a larger controlled environment, a narrower scan, or a manual review decision. What the tool should not do is consume the host because it tried to be too helpful.

This is also why streaming and limited reading matter. Large compressed files should not require the fully decompressed content to fit in memory before scanning can begin. Gzip and XZ handling now use limited readers, and archive extraction uses spooling thresholds rather than assuming every child payload belongs in memory. Nested content carries scan context so depth, child creation, and expanded bytes can be constrained across the scan rather than treated as isolated parser details. The mechanism is not complicated, but the discipline matters: decompression is not free, recursion is not free, and parsing is not automatically safe because the source is internal.

PANhunt 2.0 also improves format coverage and behaviour around common enterprise file types. It searches plain text and text-like files, RTF, legacy Office binary files, modern Office Open XML files, OpenDocument containers, PDFs, Outlook and email formats such as PST, MSG, EML, and MBOX, and recursive archive or compressed formats such as ZIP, TAR, GZ, and XZ. Access databases are listed but not yet searched. This matters because PAN rarely appears only in the neat location someone would choose for a test. It appears in exports, attachments, mail bodies, old documents, archived logs, and temporary files that stopped being temporary.

The detection model remains deliberately simple. PANhunt uses card-brand regular expressions and then validates candidates with Luhn checksum logic. That reduces obvious false positives, but it does not eliminate them. A Luhn-valid value is not automatically a real payment card. A matched value is not automatically a PCI DSS failure. Context still matters: source, location, ownership, retention purpose, masking, encryption, business process, and whether the value is production, test, synthetic, or stale data. PANhunt provides evidence for review, not judgment in place of review.

The masking behaviour is part of the same logic. The tool reports masked values rather than full card numbers. The first six and last four digits are enough for many investigation workflows, but routine output should not create a new exposure while trying to find an old one. A discovery tool that prints the sensitive data it discovers has only moved the problem.

There is one small behavioural change in 2.0 that I care about more than its size suggests. Running PANhunt without a target path or configuration file no longer starts a default root-directory scan. It prints a usage reminder and exits. Recursive discovery tools should not do the broadest possible operation because the operator missed an argument. If someone wants to scan a whole filesystem, that should be intentional. Silent ambition is a bad default for a scanner.

The packaging work follows the same practical line. PANhunt is now packaged through `pyproject.toml`, published as `panhunt`, and exposed through the `panhunt` console script. The minimum supported Python version is 3.9, which is deliberate because some enterprise Linux environments (RHEL, OEL, etc.) still make Python 3.9 a realistic baseline. A tool like this is less useful if it only works comfortably on the developer's newest workstation. It also documents the `libmagic` dependency more clearly, because file-type detection failures are not interesting problems to debug during an assessment.

PANhunt 2.0 is still a small open-source tool. My contribution was to take a utility that was already useful, then rework the parts that became fragile under real use: compressed files, nested containers, resource limits, configuration, packaging, and error handling. It is not meant to be a complete PCI solution, and I do not want to present it as one.

What it can do is more modest. It can help check whether PAN-like values exist in files and containers that are easy to forget during scoping discussions. In payment environments, those files are not always clean examples. They are often compressed, nested, old, malformed, copied, exported, or left behind after some operational task. PANhunt does not remove the need for review. It only gives the operator a safer way to start looking.
