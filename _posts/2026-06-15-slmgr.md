Microsoft mentioned the [deprecation of VBScript years ago](https://techcommunity.microsoft.com/blog/windows-itpro-blog/vbscript-deprecation-timelines-and-next-steps/4148301). Like many Windows transitions, it did not happen all at once.

That delay made sense for compatibility, but it left an odd situation behind. VBScript was already something many administrators wanted to restrict or remove, yet some ordinary Windows administration workflows still depended on `.vbs` files. Windows licensing was one of them.

Recently, Windows Insider builds started [showing PowerShell cmdlets](https://x.com/XenoPanther/status/2065529599778267502) that appear to cover `slmgr.vbs`-style functionality. That probably means Microsoft’s official replacement path is getting closer. I would expect those cmdlets to become generally available within the next year or so, although that is only my expectation. Until then, [`slmgr-ps`](https://github.com/zbalkan/slmgr-ps/) may still be useful.

This is not a new project for me. It comes from my sysadmin years. At the time, one of my hardening practices was to remove VBScript execution from managed Windows environments where possible. I disabled Windows Script Host, blocked `cscript.exe` and `wscript.exe`, and changed `.vbs` files to open with Notepad instead of running. I was not trying to make a point. I just did not want that execution path available by default.

Then I had to do normal Windows administration. `slmgr.vbs` was still the licensing tool. Office had `OSPP.vbs`. Some SCCM and MDT-related scripts also expected VBScript. I was working on Windows 7 to Windows 10 migration tasks, so activation was not a theoretical corner case. It was part of the migration. So I wrote a PowerShell script. The first version was not a module. It was a local script I used for my own work. Later, I cleaned it up enough to share as a gist. The first uploaded revision of that public gist is from [26 August 2020](https://gist.github.com/zbalkan/4ba92656a3a8387e6b220bcf8fcd5fc6/revisions), but that was already a cleaned-up version of something I had started using earlier.

The module came after that. I converted the script into [`slmgr-ps`](https://github.com/zbalkan/slmgr-ps/) and published it to the [PowerShell Gallery](https://www.powershellgallery.com/packages/slmgr-ps/), mostly so it could be installed normally instead of copied around:

```powershell
Install-Module slmgr-ps
```

The module currently exposes three public functions:

```powershell
Get-WindowsActivation
Start-WindowsActivation
Reset-WindowsActivation
```

`Get-WindowsActivation` reads activation information. `Start-WindowsActivation` handles supported activation workflows such as KMS activation, offline activation, and rearm. `Reset-WindowsActivation` handles supported cleanup operations such as uninstalling the product key, clearing the product key from registry storage, and clearing part of the KMS client configuration.

That split was intentional. I did not want every licensing action hidden behind one command. Reading activation state, starting activation, and resetting activation-related configuration are different operations. They should look different when someone runs them.

The limits matter. `slmgr-ps` is not parameter-compatible with `slmgr.vbs`. It is not feature-complete. It currently focuses on common Windows activation workflows around the selected Windows licensing product. It does not support activation-ID targeting, `all` product enumeration, arbitrary product-key input, token-based activation, Active Directory-based activation, KMS host configuration, or the full `slmgr.vbs` command surface. Even the KMS reset workflow should be described carefully. The current `-ClearKMSSettings` support clears the configured KMS host name, but full `/ckms` parity still needs more work and testing.

The current scope and gaps are documented in the [README](https://github.com/zbalkan/slmgr-ps/blob/main/README.md), including a comparison with documented `slmgr.vbs` options. I do not plan to treat this as a permanent alternative once Microsoft ships official PowerShell cmdlets broadly. When that happens, archiving this module and pointing people to the supported Microsoft path would be the correct outcome. Until then, it remains available.

If the missing features are needed by anyone, feel free to create an issue or a pull request.
